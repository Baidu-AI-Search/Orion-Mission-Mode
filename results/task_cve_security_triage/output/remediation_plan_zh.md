# 依赖漏洞修复计划 — `acme-webapp`

**计划日期:** 2026-04-11
**计划窗口:** 2026-04-11 → 2026-04-17（迭代截止 4 月 18 日）
**受众:** 工程领导层、DevOps、应用安全、SOC 2 审计准备

---

## 执行摘要

2026-04-10 对生产环境依赖的扫描在 `acme-webapp` Node.js 技术栈中发现了 **10 个 CVE**，包括 **2 个严重（P0）**、**2 个高危（P1）**、**4 个中危（P2）** 和 **2 个低危（P3）** 问题。其中 4 个 CVE 位于 **PCI DSS 范围内**（支付/认证/数据库路径），使得修复在变更管理 SLA 和 6 周后的 SOC 2 Type II 审计中都有时限要求。

**当前首要风险：**
1. **`jsonwebtoken` 签名绕过（CVE-2026-31845）** 正被**在野活跃利用**，允许未认证攻击者伪造任意用户的认证令牌，包括支付端点。
2. **`express` RCE（CVE-2026-29112）** 已有公开 PoC，可通过 HTTP 请求走私在我们的前端 Web 服务器上实现未认证远程代码执行。

**我们的应对措施：**
- **即时热修复（P0）：** 今天（4 月 11 日）切出带外热修复分支修补 `express` 和 `jsonwebtoken`，目标在 24 小时内（4 月 12 日 EOD 前）部署到生产环境——提前于常规部署窗口。
- **4 月 14 日窗口（P1）：** 在标准周一部署中发布 `pg` 和 `axios` 升级，关闭 SQL 注入和 SSRF 向量（均在 PCI 范围内）。
- **4 月 17 日窗口（P2 批次）：** 将剩余的中/低危运行时修复和所有构建工具升级批量纳入周四部署，在迭代内关闭 10 个 CVE 中的 8 个。
- **积压（P3）：** 剩余 2 个低危传递依赖问题将由团队在 90 天内自行安排；两者均已被配置默认值缓解，且大部分会被高优先级升级隐式解决。

**资源和风险态势：**
- 人力：4 名后端工程师 + 1 名 DevOps——足以并行处理 P0 验证和 P1 准备而不影响迭代承诺。
- 代码冻结：当前无冻结。
- 破坏性变更：4 次升级跨越主版本边界（`jsonwebtoken` 8→9、`axios` 0.21→1.6、`helmet` 4→7、`semver` 5→7）；每个都有下文记录的回归测试计划。`jsonwebtoken` v9 和 `axios` v1 是最高风险的升级，因其对认证和出站 HTTP 的核心地位。
- PCI/SOC 2 影响：所有 4 个 PCI 范围内 CVE 将在 4 月 14 日前关闭，远在 SLA 期限内且早于 SOC 2 审计窗口。

**底线：** 两个正被活跃利用的严重漏洞需要本周末/下周初的带外热修复；其他所有修复可顺利纳入本迭代的两个计划部署窗口，预计无客户感知的停机时间（滚动 ECS 部署）。

## 分批策略

CVE 按以下维度分组：(a) 共享升级路径/依赖耦合，(b) 优先级层级，(c) 部署窗口风险承受度。

| 批次 | CVE | 包 | 优先级 | 理由 |
|-------|------|----------|----------|-----------|
| **A — P0 热修复** | CVE-2026-31845, CVE-2026-29112 | `jsonwebtoken`, `express` | P0 | 两个正被活跃利用的核心 HTTP + 认证路径上的运行时严重漏洞。必须在 24 小时内发布。批量处理避免连续两次热修复部署。`qs`（CVE-2026-37188）是 express 的传递依赖，*预期*被此次升级自动修复；部署后验证。 |
| **B — P1 标准部署** | CVE-2026-27519, CVE-2026-30291 | `pg`, `axios` | P1 | 两者均为 PCI 范围内的高危运行时问题，7 天 SLA。在 4 月 14 日窗口一同发布，因两者都涉及后端 I/O（数据库 + 出站 HTTP）可共享集成测试通过。 |
| **C — P2 运行时加固** | CVE-2026-28003, CVE-2026-25877 | `lodash`, `helmet` | P2 | 运行时中危。Lodash 为低风险补丁升级；helmet 因主版本（CSP 中间件配置）需要关注。4 月 17 日发布。 |
| **D — P2 构建工具** | CVE-2026-33010, CVE-2026-34201 | `minimatch`, `semver` | P2 | 仅构建/CI。无生产流量风险。4 月 17 日与批次 C 同时发布以减少部署次数。 |
| **E — P3 积压** | CVE-2026-37188, CVE-2026-36500 | `qs`, `debug` | P3 | 低危，传递依赖，受默认值/配置缓解。`qs` 预期被批次 A 隐式修复；`debug` 受生产环境配置约束。在 90 天内由团队自行安排显式升级。 |

## 各批次测试要求

### 批次 A — P0 热修复
因为此次为带外发布，测试必须快速但有针对性：
- **jsonwebtoken 8.5.1 → 9.0.1（主版本）：**
  - 重新运行认证中间件单元测试（`src/auth/token.js`、`src/middleware/authenticate.js`）。
  - 验证 RS256 验签/签发仍可使用现有密钥材料（v9 取消了默认算法支持——确保算法被显式白名单化）。
  - 在暂存环境中冒烟测试登录、会话刷新、支付会话令牌签发。
  - 确认升级*前*签发的令牌不会被失效（向后兼容验证）。
- **express 4.17.1 → 4.19.2（次版本）：**
  - 运行完整 API 冒烟套件：所有 `/api/v1/*` 路由、中间件顺序、错误处理。
  - 验证 cookie/会话解析和 `body-parser` 行为（express 4.18+ 更改了 JSON 错误处理）。
  - Transfer-Encoding 走私回归测试（发送 chunked + CL 请求；确认 ALB/应用返回 400）。
- **门禁:** 暂存环境 ECS 部署，15 分钟浸泡，100% API 冒烟通过 → 通过滚动 ECS 部署升级到生产环境。

### 批次 B — P1 标准部署
- **pg 8.7.1 → 8.11.4（次版本）：**
  - 完整数据库集成测试套件（参数化查询 + 数组参数——精确的攻击面）。
  - 对暂存环境重放生产只读副本慢查询日志样本，以捕获执行计划退化。
  - 验证 `order` 和 `user` 模型 CRUD + 支付记录插入/查询。
- **axios 0.21.1 → 1.6.8（主版本）：**
  - 更新适配器/导入以应对任何破坏性变更（TypeScript 类型、默认 `withCredentials`、`axios` Promise 拦截器签名）。
  - 测试外部 API 流程：支付提供商 webhook 回调、地址验证、物流 API。
  - 必须使用暂存环境中的恶意重定向目标显式重新测试重定向跟随行为，确认 SSRF 已缓解。
  - 验证超时/重试中间件在新 API 下正常工作。
- **门禁:** 完整 CI + 30 分钟暂存浸泡；在 4 月 14 日窗口部署，值班 SRE 就位。

### 批次 C — P2 运行时加固
- **lodash 4.17.20 → 4.17.25（补丁）：** 低风险——运行 `config.js` 和 `data-transform.js` 的单元测试。
- **helmet 4.6.0 → 7.1.0（主版本）：**
  - helmet v7 拆分了部分默认头并更改了 CSP nonce API。
  - 在暂存环境中使用浏览器开发工具验证 HTML 服务路由的 CSP、HSTS、X-Frame-Options 和 Referrer-Policy 头仍正确设置。
  - 确认 API 路由（不设置 HTML CSP）不受影响。
- **门禁:** 标准 CI；在暂存环境中目视检查安全头。

### 批次 D — P2 构建工具
- **minimatch 3.0.4 → 3.1.2（次版本）：** 重新运行构建脚本，测试打包/lint 中使用的 glob 模式。
- **semver 5.7.1 → 7.5.4（主版本）：**
  - semver v7 更改了部分范围解析边缘情况。验证 CI 版本标记、发布脚本，以及任何解析 semver 范围的运行时代码（lockfile/版本门控）。
- **门禁:** 干净安装后 CI 全绿；无生产部署风险（仅 devDependencies）。

### 批次 E — P3 积压
- **qs / debug:** 批次 A 上线后，运行 `npm ls qs` 和 `npm ls debug` 查看传递解析是否已引入修复版本。若未修复，在 90 天窗口内按团队节奏在 `package.json` 中添加显式 `resolutions`（或 `overrides`）。
- 配置审计（零成本缓解，立即执行）：验证 Fargate 任务定义未在生产环境设置 `DEBUG=*`。

## 主版本风险登记表

以下升级跨越主版本边界，回归风险最高。列出缓解措施供审查者重点关注：

| 包 | 变更 | 风险领域 | 缓解措施/审查者检查清单 |
|---------|--------|------------|---------------------------------|
| **jsonwebtoken** | 8.5.1 → 9.0.1 | 算法混淆是*该*漏洞类别；v9 要求在 `verify()` 中显式指定 `algorithms`。现有 RS256 令牌必须仍可验证。回调/Promise API 变更。 | 审计每个 `jwt.verify` 调用确保显式 `algorithms: ['RS256']`；回归测试令牌签发 + 刷新；确认旧会话未被失效。 |
| **axios** | 0.21.1 → 1.6.8 | `withCredentials`、CSRF/XSRF 头名称、适配器解析、最大重定向数和错误形态的默认值已变更。Webhook 签名验证可能依赖响应缓冲。 | 搜索 `axios.create` 默认值；在暂存环境重放记录的生产 webhook 负载；使用控制的 30x 到内部 IP 测试重定向跟随（应失败关闭）。 |
| **helmet** | 4.6.0 → 7.1.0 | `helmet.contentSecurityPolicy()` 默认值已变更；部分中间件被拆分到独立包；基于 nonce 的 CSP 指令排序已变更。 | 对比升级前后的响应头基线；验证每请求 CSP nonce 轮换；确保 HSTS `max-age` 保留。 |
| **semver** | 5.7.1 → 7.5.4 | 仅构建工具；semver v7 默认取消松散模式解析并更改部分预发布版本比较。 | 在临时 git 标签上重新运行发布/标记脚本；确认 CI 正确解析 lockfile 中的依赖范围。 |
| **debug** | 2.6.9 → 4.3.5 | 传递依赖；v3+ 更改了命名空间格式和环境变量处理。仅在启用 DEBUG 时有影响（当前未启用）。 | 不在批次 A–D 中主动升级；确认生产环境干净（已为 P3 控制措施）。 |

次版本/补丁升级（`express`、`pg`、`lodash`、`minimatch`、`qs`）根据上游变更日志和标准 SemVer 保证为低风险；现有测试的回归覆盖率已足够。

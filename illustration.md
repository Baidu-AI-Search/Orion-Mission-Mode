# PinchBench 自动检查适配说明
本文档记录了百度 AI 搜索任务化模式在 PinchBench 评测中的适配工作。在完整覆盖的 147 个任务中，我们共对26个任务的评分脚本进行了必要的适配调整。



**适配的核心理由：** 现有评测框架的自动化评分脚本在格式匹配等层面存在若干刚性假设，而大模型 Agent 的输出格式具有天然的多样性——这导致部分任务出现"结果正确却被判错"的情况。为还原真实能力水平，我们对评分脚本进行了修正，而非降低评测标准。所有适配均遵循严格原则，确保评测的公平性与分数的可信度。

---

## 适配原则
在适配过程中严格遵守以下两条底线，绝不通过放宽标准来换取分数：

1. **不通过虚假结果注水 —** 如果 Agent 确实未完成某项检查、答案存在事实错误或关键能力缺失，适配后的评分脚本仍然判定为失败，绝不将错误结果判为正确。
2. **不改变原始评分标准 —** 所有数值阈值、关键词列表、评分档位、评判维度均与官方基准完全保持一致，不降低任何一个检查项的通过门槛。

适配仅针对"结果正确但被判错"的场景进行修复，确保评分能真实反映 Agent 的实际能力。

---

## A 类 · 任务执行**轨迹来源适配**
* **问题背景：**
    * 原始评分脚本期望 Agent 的执行轨迹（调用了哪些工具、执行了什么命令）通过 `transcript` 参数传入。但我们的系统架构中，轨迹不是存在这个参数里的，而是记录在独立的执行日志中。
    * **这种不匹配并非能力问题**——Agent 实际完成了所有任务，内容完整正确，只是执行轨迹来源差异导致评分脚本无法读取到内容。若不进行适配，这类任务会被误判为"未生成执行轨迹"，与事实严重不符。

* **修复方式**
    * 从 Agent 的工具调用记录中还原文件内容，将其作为评分输入传入原评分逻辑。
    * 评分逻辑本身完全不变，只是补齐了证据来源，让评分脚本能看到 Agent 真正产出的成果。

* **典型案例task_id=**`task_cicd_pipeline_debug`
    * Agent 通过 `write` 工具成功写入了自动化脚本与分析笔记文件，内容完整且符合任务要求。但这些文件存储在 Agent 运行时的独立工作目录中，评分脚本在默认 workspace 路径下找不到对应文件。
    * Agent 通过编辑命令正确修复了 `ci.yml` 中的 bug，但评分脚本因 `transcript` 为空而看不到这些操作。适配后改为从执行日志读取，最终正确反映了 Agent 的完成情况。

## B 类：**语义等价但格式不同的兼容**
* **问题背景：**
    * 大模型 Agent 在输出正确结果时，往往会采用更符合人类阅读习惯、更结构化的呈现方式。但部分评分脚本使用了刚性的字符串匹配或正则模式，仅接受特定的书写格式——即使语义完全等价、数值完全一致，仅因排版格式不同就会被判定为错误或扣分。
    * **这种扣分显然不合理**——评测的目标是检验 Agent 是否具备解决问题的能力，而非考核其是否严格遵循某一种特定的输出排版。对语义正确的等价写法给予认可，是保证公平的修正。

* **修复方式：**
    * 调整匹配模式，使其能接受同一正确值的多种等价写法，但**绝不引入新的可接受值**——即不会把原本错误的答案放宽为正确，只是让格式不再成为正确答案被误杀的原因。

* **典型案例****task_id=**`task_log_hdfs_slow_ops`**：**
    * 评分脚本要求精确匹配字符串 `"91178"`，Agent 输出的是带千分位分隔符的 `"91,178"`。两者指向完全相同的数值，但字符串不同导致匹配失败。
    * 适配后，评分脚本在提取数值时会先去除千分位分隔符再进行比对——底层数值完全一致，不应因格式差异扣分。
